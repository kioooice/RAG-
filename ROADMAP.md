# 能力验证路线

路线由能力缺口和验收证据驱动，不再按算法、框架或 Notebook 课程顺序推进。统一决策顺序是：成熟方案优先 → 必要时受控比较 → 调整边界与规则 → 真实缺口出现后才自定义开发。

## 已完成：证据基线

### 001 检索对照基线

- 保留 TF-IDF、固定 Corpus、Validation/Test 指标和失败样本，作为后续检索方案的最低对照。
- 它证明评测链路可运行，不代表已经完成知识库检索或完整 RAG。
- 结果保持不变；后续方案必须在相同或明确可比的条件下给出证据。

### 002 文档测试基线

- 保留虚构多格式测试包生成、格式路由、统一知识格式、质量检查和确定性验证。
- 它是 Document Intelligence 候选方案的对照测试基线，不是生产解析系统。
- 扫描 PDF 和图片只验证了 `needs_ocr` 路由，没有实现 OCR。

## 下一阶段：Document Intelligence 成熟方案筛选与集成

目标是选择并接入能够可靠处理目标资料的成熟方案，而不是从头实现 OCR 或文档解析引擎。

1. 用统一能力卡确定真实资料范围、隐私边界、部署约束和验收标准。
2. 筛选少量有维护、有明确许可且符合运行约束的成熟候选方案。
3. 使用 `002_ingestion_baseline` 的受控材料进行同条件比较。
4. 比较格式覆盖、文本和表格保真、版面与来源定位、OCR、稳定性、成本、许可、可部署性及可追溯性。
5. 选择方案后只做最小集成，明确输入输出契约、异常路由和人工复核边界。
6. 用测试报告验收；若不达标，先调整边界、配置或候选组合。

停止条件：已有成熟方案通过约定验收，即停止自研。只有可复现证据证明关键缺口无法通过成熟方案、配置或组合解决，并经用户确认后，才提出局部自定义开发。

## 后续能力区域

后续顺序不预先锁死，由 `D:\Projects\study\AI_CAPABILITY_ATLAS.md` 中的能力卡和实际业务缺口决定：

- 检索与知识访问：以 `001` 为对照筛选词法、向量、混合检索和重排方案。
- 生成与推理：验证有依据回答、上下文边界、工具使用和拒答行为。
- 评测与可观测性：建立离线评测、运行追踪、成本、延迟与失败归因。
- 模型适配与优化：只有通用模型存在稳定、可量化缺口时才评估微调或量化。
- 系统集成与治理：验证权限、隐私、安全、工作流、部署和人工控制。

## 每张能力卡的决策门

- 是否存在明确、重复、值得解决的真实问题？
- 是否已有成熟方案满足验收标准？
- 是否必须比较，比较会改变什么决定？
- 能否只通过边界、规则、配置或组合解决？
- 自定义开发的缺口证据、最小范围和停止条件是什么？

所有实验继续记录数据版本、候选方案、配置、指标、运行时间、成本、失败信息和决策结论；不得使用测试集调参。
