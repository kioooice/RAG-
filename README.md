# Retrieval Adaptation Lab

这是一个以真实证据验证 AI 系统能力的实验仓库。学习重点不是重复手写成熟技术，而是学会定义目标、选择方案、设定边界、验收结果，并在证据不足时纠偏。

## 学习模式

- 用户是 AI 系统的控制面：负责目标、约束、方法选择、验收标准和纠偏。
- Codex 是执行面：负责环境、方案调研与集成、代码、实验、验证和证据整理。
- 成熟方案优先；只有选择会实质影响结果时才做受控比较。
- 优先调整输入输出边界、规则、配置和验收标准。
- 只有真实缺口被实验明确证明，且成熟方案无法满足时，才进入自定义开发。

本项目不再把以下内容作为学习目标：

- 为练习而手写底层算法；
- 按课程顺序逐个完成 Notebook；
- 重复实现已有稳定工具或框架；
- 仅按技术名词清单安装组件，而没有明确能力目标和验收证据。

成熟方案可以是黑盒，但它的输入输出、限制、成本、许可、失败模式和验收结果必须可观察。学习对象从“内部每一行如何实现”转为“系统边界是否正确、能力是否真实成立”。

## 项目位置与存储规则

- 当前正式项目位于 `D:\Projects\study\retrieval-adaptation-lab`。
- C 盘原目录仅作为迁移前副本保留，不再写入新内容，也不得删除。
- Hugging Face、Torch 与 pip 缓存统一写入 `D:\AI-Lab\cache`；大型数据与模型分别写入 `D:\AI-Lab\data` 和 `D:\AI-Lab\models`。
- 下载数据集或模型前必须报告预计大小；模型权重、数据集缓存、本地环境和密钥不得提交 Git。
- 项目脚本应从自身位置推导项目根目录，避免写死用户目录。

## 目录说明

- `iterations/`：能力基线、指标、失败样本和结论。
- `src/`：已被实验需要的可复用实现，不按技术清单预建模块。
- `experiments/`：配置、结果和实验报告。
- `scripts/`：环境、复现和验证入口。
- `notebooks/`：历史认知材料或临时分析，不是默认课程路线。
- `data/`、`models/`：本地数据或模型位置；大型内容不提交 Git。

新增依赖或技术实现前，必须先说明它解决的能力缺口、成熟替代方案和验收方式。
