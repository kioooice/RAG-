# AGENTS.md

## 项目定位与角色

1. 本项目是 AI 系统能力验证实验室，不是 Notebook 课程、算法手写练习或成熟技术复刻项目。
2. 用户是控制面，负责目标、约束、方法选择、验收标准和纠偏；Codex 是执行面，负责环境、候选方案调查与集成、代码、实验、验证和证据整理。
3. Codex 必须把选择依据、系统边界、配置、失败模式和验证证据呈现给用户，不以“工具运行成功”替代能力验收。

## 方案选择规则

4. 成熟方案优先；先检查现有工具、服务、框架或合理组合，再考虑自定义实现。
5. 只有比较结果会影响选型时才开展受控比较，并提前明确比较维度和停止条件。
6. 优先通过输入输出契约、规则、配置、异常路由和人工复核边界解决问题。
7. 只有可复现实验证明成熟方案不能满足关键验收标准，且用户确认缺口值得解决时，才允许局部自定义开发。
8. 不以手写底层算法、逐课完成 Notebook、重复实现成熟技术或堆叠工具作为学习成果。
9. 新增依赖或技术实现前，说明它对应的能力缺口、成熟替代方案、资源成本和验收方式。

## 基线与当前方向

10. `iterations/001_retrieval_baseline` 是检索对照基线；`iterations/002_ingestion_baseline` 是文档测试基线。保留既有结果、失败信息和实验边界，不因路线调整而改写。
11. Notebook 是历史认知材料或临时分析载体，不是默认学习主线；未经用户确认不得删除遗留 Notebook。
12. 下一阶段只做 Document Intelligence 成熟方案筛选、对照验证和最小集成，不独立从头实现 OCR。

## 执行与验收方式

13. 开始前读取能力卡或明确目标、输入、约束、验收指标和不做事项。
14. 先盘点成熟候选，再执行满足决策需要的最小实验；不要预建未来可能使用的复杂架构。
15. 集成后必须保留来源追溯、错误与警告、实际成本、版本和可复现命令。
16. 每次实验只改变一个主要变量；训练、验证、测试职责必须明确，不得使用测试集调参。
17. 错误记录需要保留，不能只留下最终成功结果。
18. 给用户的总结使用初学者可理解的中文，重点报告能力是否成立、证据边界、风险和下一项决策。

## 环境与安全

19. Python 依赖只能使用项目根目录的 `.venv`，不得全局安装。
20. 系统级安装、管理员操作、删除或覆盖已有文件必须先获得确认。
21. 不得把密钥、Token 和密码写入代码、Notebook、配置或日志。
22. 长时间训练或大型下载前，先估算显存、内存、磁盘、时间和下载量；单项预计超过 1 GB 时必须暂停确认。
23. 不执行破坏性 Git 操作，不擅自连接远程仓库或推送。
24. 当前正式项目位于 D 盘 `study` 工作区；C 盘旧目录仅为迁移前副本，不得继续写入或删除。
25. 大型数据、模型和 Hugging Face、Torch、pip 缓存必须写入 `D:\AI-Lab` 下的对应目录。
26. 下载数据集或模型前必须报告预计大小；模型权重、数据集缓存、本地环境和密钥不得提交 Git。
27. Notebook 需要运行时优先通过 `scripts\start_lab.ps1` 启动；项目脚本优先使用相对路径，不得写死用户目录。
28. 每个迭代完成后更新 `PROJECT_STATE.md`，记录最后提交、能力状态、下一步和未解决问题。

## 修改纪律

- 先读取现有状态，再做最小、安全、可验证的修改。
- 不修改与当前能力卡无关的实验结果或用户工作区改动。
- 完成后报告修改文件、验证方式、未覆盖边界和是否需要用户决策。
